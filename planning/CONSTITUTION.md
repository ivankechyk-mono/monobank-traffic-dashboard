## Принципи якості

### 1. Secrets — тільки через змінні оточення
OAuth tokens, client_secret.json, API keys та будь-які credentials НІКОЛИ не потрапляють у код або git-репозиторій. Всі секрети читаються виключно через `os.environ` або `.env`-файл, який додано до `.gitignore`. Перед кожним commit — перевірка `git-secrets` або `truffleHog`. Порушення блокує merge.

### 2. Read-only доступ — без виключень
Всі Google API scope'и залишаються read-only: `analytics.readonly`, `webmasters.readonly`, `adwords` (тільки читання звітів). Запис у джерела даних заборонено на рівні OAuth scopes. Сирі персональні дані користувачів (user-id, IP, PII) не зберігаються і не логуються — тільки агреговані метрики.

### 3. Postgres скрізь де потрібна БД
Якщо потрібне постійне зберігання (кеш метрик, знімки даних, scheduling) — виключно PostgreSQL. SQLite заборонено. Локально: `docker-compose` з `postgres:16`. Продакшн: керована БД (Cloud SQL або Supabase). Жодних in-memory-only рішень для даних що мають переживати рестарт.

### 4. Скрипти → сервіс: чіткий поділ шарів
Код організований у три шари без змішування:
- `connectors/` — тільки виклики Google API, повертають сирі dict/DataFrame
- `transforms/` — тільки бізнес-логіка обробки, без I/O
- `loaders/` — тільки запис у Postgres або Looker Studio

Один модуль — одна відповідальність. Скрипти-"комбайни" (fetch + transform + write в одному файлі) заборонені.

### 5. Тести покривають критичний шлях
Unit-тести обов'язкові для `transforms/` (бізнес-логіка, розрахунки метрик) — мінімум 80% покриття. Інтеграційні тести для `connectors/` запускаються з mock-відповідями Google API (без реальних викликів в CI). Кожен новий connector має хоча б один smoke-тест що перевіряє структуру відповіді.

## Архітектурні обмеження

- Python 3.11+. Без Node.js, без Go — тільки Python для всього backend-коду.
- Looker Studio як єдиний шар візуалізації. Власний frontend не будуємо.
- Планувальник задач — тільки через `cron` або Cloud Scheduler. Без Celery, без RQ для MVP.
- Залежності фіксуються у `requirements.txt` з точними версіями (`==`). `>=` заборонено у production-файлі.
- Docker обов'язковий для локального запуску. "Запускається тільки на моїй машині" — не прийнятно.

## Тестування

- Unit тести: `pytest` для всього в `transforms/`. Покриття ≥ 80% (перевіряється в CI через `pytest-cov`).
- Інтеграційні тести: connectors тестуються з `unittest.mock` або `pytest-httpx`. Реальні API-виклики в тестах заборонені.
- Smoke-тест pipeline запускається після кожного деплою і перевіряє що дані за вчора успішно завантажились у Postgres.
- Тести живуть у `tests/` із дзеркальною структурою відносно `src/`.

## Безпека

- `.gitignore` обов'язково включає: `*.json` (credentials), `.env`, `token.pickle`, `token.json`, `client_secret*.json`.
- `client_secret.json` не комітиться ніколи. Замість нього — path до файлу через env-змінну `GOOGLE_CREDENTIALS_PATH`.
- Логи не містять значень токенів, ключів або PII. Логується тільки `token_type`, `expiry`, `scopes` — без `access_token` і `refresh_token`.
- Ротація токенів: OAuth refresh виконується автоматично через `google-auth` library. Ручне зберігання токенів у БД заборонено.
- Доступ до Postgres — тільки через SSL у продакшні (`sslmode=require`).

## Governance

Кожна зміна фіксується з датою, причиною і іменем автора.
